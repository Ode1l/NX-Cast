# NX-Cast 桌面快捷方式方案

## 背景

`NX-Cast` 当前以标准 Homebrew `NRO` 形式运行，典型启动方式是进入 `hbmenu` 后从相册或游戏接管模式打开。

目标是让 `NX-Cast` 像 `wiliwili` 一样，支持从 Switch 桌面直接点击图标进入应用，而不是每次先进 `hbmenu`。

这里的“桌面快捷方式”本质上不是把 `NRO` 直接固定到桌面，而是安装一个 `NSP forwarder`：

1. 桌面显示为一个已安装应用图标
2. 点击后启动一个很小的 forwarder
3. forwarder 再去 SD 卡上查找并启动真正的 `NRO`

## 外部参考结论

从 `wiliwili` 项目公开文档可以确认：

1. `wiliwili` 的 Switch 版本提供“在应用内安装桌面图标”的能力，入口在“设置 / 实用工具 / 使用教程”。
2. 它的桌面图标优先尝试启动：
   - `sd:/switch/wiliwili.nro`
   - `sd:/switch/wiliwili/wiliwili.nro`
3. 如果两个路径都不存在，则退回到 `hbmenu` 手动选择。
4. `wiliwili` 在后续版本里还专门修过一次 “Switch 19.0.0+ 系统下桌面图标崩溃”，并明确提示需要重新安装桌面图标。

这说明 `wiliwili` 采用的不是“系统帮你固定 NRO”，而是标准的 `NRO -> NSP forwarder` 路线，而且该路线会受 Switch 固件版本影响。

## 对 NX-Cast 的含义

`NX-Cast` 如果要实现同类能力，正确方向不是修改当前 `NRO` 启动方式本身，而是补一层 forwarder 发行与安装能力。

也就是：

1. 保留 `NX-Cast.nro` 作为真实程序主体
2. 增加一个 `NX-Cast` 专用桌面图标 `NSP`
3. 让该 `NSP` 固定查找 `NX-Cast.nro`
4. 用户安装一次后，日常从桌面启动

## 推荐实现路线

推荐分两阶段做。

### 第一阶段：先把“桌面快捷方式”定义成发行物，而不是应用内即时生成

先不要在 `NX-Cast` 应用里集成 `NSP` 生成器，也不要一开始就在设备端做安装流程。

先做最小可行方案：

1. 保持现有 `NX-Cast.nro` 构建链不变
2. 额外产出一个预构建的 `NX-Cast-forwarder.nsp`
3. 文档规定 `NRO` 的固定放置路径
4. 发布时同时提供：
   - `NX-Cast.nro`
   - `NX-Cast-forwarder.nsp`
   - 安装说明

这样做的优点：

1. 不侵入当前运行时代码
2. 风险集中在发行链，不会把安装器逻辑混进播放器主程序
3. 更容易先验证路径、图标、Title ID、固件兼容性

### 第二阶段：再决定是否做应用内入口

等第一阶段稳定后，再考虑在 `NX-Cast` 里做：

1. “使用教程”页面
2. “检测桌面图标是否已安装”
3. “提示重新安装 forwarder”

不建议一开始就做“应用内直接生成 NSP 并安装”。

原因：

1. 这会把打包、签名、安装、权限、固件适配都拉进应用内
2. 对当前项目主线价值不高
3. 一旦 Switch 固件升级，维护成本会明显上升

## 建议的 forwarder 行为

建议 `NX-Cast` 的桌面图标按固定顺序查找真实 `NRO`：

1. `sd:/switch/NX-Cast.nro`
2. `sd:/switch/NX-Cast/NX-Cast.nro`
3. 找不到时回退到 `hbmenu`

这样做的原因：

1. 和 `wiliwili` 的用户心智一致
2. 同时兼容“单文件放在 switch 根目录下”和“独立目录分发”两种发布方式
3. 便于未来接入 HB App Store 一类工具时保留兼容性

## 仓库内需要新增的内容

如果按第一阶段推进，仓库建议新增这些内容。

### 1. 文档

新增面向用户的安装文档，至少说明：

1. `NRO` 应放到哪个路径
2. `forwarder NSP` 如何安装
3. 固件升级后如桌面图标失效，需要重新安装
4. 图标只负责跳转到 SD 卡上的 `NRO`，更新应用通常只需替换 `NRO`

### 2. Forwarder 构建目录

建议新增一个独立目录，例如：

```text
forwarder/
  README.md
  icon.jpg
  config/
  scripts/
```

这个目录只负责桌面图标相关资源，不与主程序源码混在一起。

### 3. 路径约定

在文档和发行流程里固定：

1. 首选 `sd:/switch/NX-Cast.nro`
2. 兼容 `sd:/switch/NX-Cast/NX-Cast.nro`

路径一旦发布，后续尽量不要改。

## 构建方式建议

这里不建议自己从零写 forwarder ROM，直接复用现有成熟工具链更实际。

可选方案：

1. 用 `NTON` 在 PC 侧生成 `NRO -> NSP forwarder`
2. 用 `switch-nsp-forwarder` 的现成 forwarder 方案
3. 需要更深度定制时，再研究它们引用的 ROM / exefs 模板

对 `NX-Cast` 来说，优先级建议是：

1. 先验证 `NTON` 或等价工具能否稳定生成可用 NSP
2. 再决定是否把这条流程脚本化到本仓库

## 与当前仓库的关系

当前仓库已经有：

1. `icon.jpg`
2. `APP_TITLE / APP_AUTHOR / APP_VERSION`
3. `.nro / .nacp` 产物

当前仓库还没有：

1. 专用桌面图标 `NSP` 产物
2. forwarder 的固定 `Title ID` 管理
3. 面向用户的桌面图标安装说明
4. 固件升级后的 forwarder 兼容策略

所以现阶段更像“已经具备素材，但还没有桌面图标发行链”。

## 风险与注意事项

### 1. 固件兼容风险

这是这类功能最大的现实问题。

已知公开工具和 `wiliwili` 都遇到过因为系统固件升级导致 forwarder 失效的问题，因此该能力需要视作“与系统版本耦合的发行物”，不能把它当成一次实现永久稳定。

### 2. Title ID 冲突

需要为 `NX-Cast` 的 forwarder 选择独立且不冲突的 `Title ID`。这个值应该固定管理，不能每次临时生成。

### 3. 更新策略

推荐把更新策略设计成：

1. 平时更新只替换 `NRO`
2. 只有 forwarder 失效或图标元数据变化时才重新安装 `NSP`

### 4. 安装失败与回退

即使未来做应用内入口，也应保留最基础回退方案：

1. 用户仍可从 `hbmenu` 直接打开 `NX-Cast.nro`
2. 桌面图标只是增强体验，不应成为唯一入口

## 建议的最小交付

第一版建议只交付这些内容：

1. 固定 `NRO` 路径策略
2. 一个可安装的 `NX-Cast-forwarder.nsp`
3. 一份用户安装说明
4. 一份开发维护说明

不建议第一版就做：

1. 应用内自动生成 forwarder
2. 应用内自动安装 NSP
3. 多路径动态选择 UI
4. 复杂的图标安装检测逻辑

## 建议的后续开发顺序

1. 先确定 `NX-Cast.nro` 的最终放置路径约定
2. 选定 forwarder 工具链
3. 手工生成一个可安装的 `NX-Cast-forwarder.nsp`
4. 在真实设备上验证：
   - 桌面启动
   - NRO 路径回退
   - 替换 NRO 后继续可用
5. 验证通过后再把这套流程脚本化
6. 最后才考虑在应用里增加“使用教程 / 桌面图标说明”

## 对当前项目的最终建议

对 `NX-Cast`，最合理的落地方式是：

1. 采用 `wiliwili` 同类的 `NSP forwarder -> NRO` 架构
2. 第一阶段只做“预构建 forwarder + 文档 + 路径约定”
3. 不把 forwarder 生成和安装逻辑直接塞进当前应用主程序

这样可以先获得“桌面一键启动”的用户体验，同时把风险控制在发行链，而不是播放器运行时。
