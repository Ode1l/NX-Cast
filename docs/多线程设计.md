# 多线程设计

本文档整理了当前对 NX-Cast（DLNA DMR + 后续多协议）线程模型的讨论结论。

## 1. 目标与约束

- 项目重点不是高并发吞吐，而是协议之间的正确性和可维护性。
- 需要保证：同一时刻只有一个协议会话控制播放器，不出现状态错乱。

## 2. MVP 线程模型（当前建议）

- `SSDP线程`（1个）
  - 只负责 UDP 1900 发现收发（`M-SEARCH` 响应 / 可选 `NOTIFY`）。
- `HTTP控制线程`（1个）
  - 统一处理 HTTP 接入：`device.xml`、`/scpd/*.xml`、`/upnp/control/*`（SOAP）。
  - `SOAP` 作为 HTTP 路由，不单独再拆线程。
- `播放线程`（1个）
  - 负责拉流、解码、渲染。
  - 使用状态机驱动播放行为。

## 3. 播放状态机

建议最小状态：

- `IDLE`
- `READY`
- `PLAYING`
- `PAUSED`
- `STOPPED`
- `ERROR`

状态变更原则：

- 只允许播放线程修改播放状态。
- 协议线程（DLNA/AirPlay）通过消息队列发送 `SetURI/Play/Pause/Stop/Seek`，不直接改状态。

## 4. 多协议扩展设计

未来支持 AirPlay 等协议时，不建议“每个协议一套播放线程”，建议结构：

- 各协议各自有控制入口（可独立线程）。
- 中间增加 `会话仲裁层`（ownership arbitration）。
- 播放层仍保持一个统一播放线程（协议无关）。

## 5. 控制权与抢占规则

建议固定三条规则：

- 单一播放所有权：`owner = NONE | DLNA | AIRPLAY`
- 状态单写：播放状态只由播放线程写
- 显式抢占策略（不要隐式覆盖）：
  - `reject_if_busy`（MVP 推荐，最稳）
  - 或 `force_preempt`（先 stop 旧会话，再切新会话）

建议补充会话字段：

- `owner_protocol`
- `session_id`
- `generation`（每次切会话递增，旧命令自动失效）

## 6. 并发关系说明

- `SSDP` 不会直接调用 `SCPD/HTTP` 代码路径。
- 实际流程是：控制端先通过 SSDP 拿到 `LOCATION`，随后再发 HTTP 请求读取 `device.xml/SCPD/SOAP`。
- 因此是“并行发生”，不是“函数调用嵌套”。

## 7. 线程间通信建议

- `HTTP控制线程 -> 播放线程`：命令队列（`SetURI/Play/Pause/Stop/Seek/SetVolume/SetMute`）。
- `播放线程 -> HTTP控制线程`：事件队列（`State/Position/Duration/Volume/Mute/Error`）。
- 对外可见的协议状态由 `HTTP控制线程` 单点维护，避免多线程直接写同一状态。

## 8. 关于 `select()`

`select()` 是 I/O 多路复用：一个线程同时等待多个 socket 的可读/可写事件。

- `ret > 0`：有 fd 就绪
- `ret == 0`：超时
- `ret < 0`：错误

可用于 `SSDP线程` 和 `HTTP控制线程` 内部的 socket 轮询，保持实现简单可控。

## 9. 当前结论（一句话）

先落地“3线程（SSDP + HTTP控制 + Player）+ 队列通信 + 统一播放状态机 + 会话所有权仲裁”，再在此基础上平滑扩展多协议。

## 10. 当前实现状态（2026-03-30）

### 10.1 当前实现能保证什么

当前代码还没有完全落地第 9 节描述的“正式线程模型”，但已经具备最小并发保护：

1. `SSDP` 已有独立线程
2. `HTTP/SOAP` 已有独立线程
3. `player_backend_libmpv` 内部增加了 `Mutex`，可避免 `player` 内部状态的直接数据竞争
4. 当前 `HTTP server` 仍是单线程串行处理请求，因此现阶段大多数 SOAP 控制命令不会并发打进 `player`

这意味着：

1. 当前实现可以支撑 Step 1 的 MVP 目标
2. `SetURI / Play / Pause / Stop / Seek / Volume / Mute` 已能绑定到真实 `libmpv`
3. 当前不会因为最基础的多线程读写而立即出现内存破坏

### 10.2 当前实现还不能算“完整多线程设计”

现在的设计只能说“有基本互斥保护”，不能说“多线程模型已经完成”。

根本原因是：

1. `player` 还没有独立 backend 线程
2. 还没有命令队列
3. 还没有事件队列
4. 还没有会话仲裁层
5. 还没有对 `mpv` 事件流做专门线程化处理

也就是说，当前 `player` 仍然是：

1. SOAP 线程直接调用 `player_*`
2. `player` 再直接调用 `libmpv`
3. `Mutex` 只负责保护临界区，不负责解决更高层的控制时序问题

## 11. 当前存在的问题

### 11.1 播放命令仍运行在 SOAP/HTTP 线程里

当前 `SetURI / Play / Stop / Seek` 实际是在调用者线程里执行的，而不是投递到独立播放线程。

这会带来两个问题：

1. 如果 `mpv_command()` 或属性访问阻塞，HTTP/SOAP 响应也会被一起阻塞
2. 后续一旦播放打开、网络拉流、状态刷新变重，就会拖慢控制层响应

这不是数据竞争问题，而是“线程职责没有解耦”的问题。

### 11.2 没有独立的 mpv 事件循环

当前 `libmpv` backend 主要靠：

1. 动作执行后主动刷新状态
2. 查询时读取属性

但还没有：

1. `mpv_wait_event()` 专用线程
2. `mpv_observe_property()` 持续监听
3. `FILE_LOADED / END_FILE / PROPERTY_CHANGE / ERROR` 的异步事件处理

结果是：

1. 有些状态变化只能在“下一次命令”或“下一次查询”时才被看见
2. `buffering / end-of-stream / loading failed` 这类事件还不能稳定地主动推送给控制层

### 11.3 事件回调仍在持锁状态下触发

当前 `player_backend_libmpv` 会在持有内部 `Mutex` 时直接调用事件 sink。

现在之所以还能工作，是因为当前事件接收方只更新 `g_soap_runtime_state`，不会立刻反调 `player_*`。

但这个结构有明确风险：

1. 如果未来事件处理路径里又调用 `player_*`
2. 或者事件接收方做更复杂逻辑

就可能形成重入或死锁风险。

因此，正式方案里应改为：

1. backend 线程只产生命令结果或状态快照
2. 通过事件队列/消息队列再交给控制层
3. 避免“持锁调用外部回调”

### 11.4 没有会话所有权仲裁

当前系统还没有实现：

1. `owner_protocol`
2. `session_id`
3. `generation`
4. `reject_if_busy / force_preempt`

因此现在只能说：

1. 单一控制路径下基本可用
2. 多协议并存或多控制端抢占时，逻辑顺序还不可靠

即使加了 `Mutex`，也只能保证“不会同时写坏内存”，不能保证“命令语义一定正确”。

### 11.5 对外运行时状态还没有统一加锁策略

`g_soap_runtime_state` 现在主要通过事件回调和控制路径更新。

在当前实现里，这还能工作，原因是：

1. `HTTP server` 目前是单线程串行处理
2. `player` 还没有真正独立事件线程

但一旦后续引入：

1. 独立 player 线程
2. 独立 mpv 事件线程
3. 多协议控制入口

就必须明确：

1. `g_soap_runtime_state` 由谁单写
2. 其他线程如何读
3. 是加锁，还是改成单线程 owner + 队列同步

否则会出现控制层状态和真实播放状态不同步的问题。

## 12. 当前结论

### 12.1 当前实现是否“能处理好多线程”

结论：不能说“已经处理好”，只能说“已经做了第一层保护”。

更准确的表述是：

1. 当前实现可以承受 Step 1 范围内的基础并发
2. 当前实现不能视为最终多线程设计完成

### 12.2 当前最主要的缺口

必须补的：

1. 独立 player backend 线程
2. 命令队列
3. mpv 事件线程或等价事件循环
4. 事件队列
5. 会话仲裁层
6. 运行时状态的统一写入模型

可后续补的：

1. 更细粒度的状态分类（`OPENING / BUFFERING / SEEKING / ENDED`）
2. 多协议抢占策略
3. 更复杂的事件订阅机制

### 12.3 对 Step 1 的判断

Step 1 的目标是“把 DLNA 控制动作绑定到真实 `libmpv`”。

按这个目标看，当前实现是成立的。

但如果目标变成：

1. 稳定长期运行
2. 真正异步播放
3. 多协议并行控制
4. 真实渲染与播放线程协作

那就必须进入 Step 3，把正式线程模型补完。
