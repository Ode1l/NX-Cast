# 多线程设计

本文档只讨论当前 `NX-Cast` 的并发模型。

## 1. 目标

当前线程模型要保证：

1. 协议入口不直接阻塞在播放重操作上
2. 播放状态只有一个真实写入来源
3. 协议、播放、事件推送边界清楚

## 2. 当前线程组成

当前稳定存在这些执行单元：

1. `main` / 前台渲染主循环
2. `SSDP` 线程
3. `HTTP` 线程
4. `player owner` 线程
5. `GENA event` worker

## 3. 当前并发模型

```text
HTTP / SOAP
  -> player command queue
  -> player owner thread
  -> backend
  -> player snapshot / event
  -> protocol_state
  -> SOAP query / GENA notify
```

## 4. player owner thread

当前 owner thread 是播放控制唯一执行者。

它负责：

1. 打开媒体
2. 播放控制
3. 处理 backend 事件
4. 更新 snapshot
5. 发出 `PlayerEvent`

这保证：

1. `SOAP` 线程不会直接打 backend
2. 播放命令顺序可控
3. backend ownership 清楚

## 5. protocol_state 的位置

`protocol_state` 不是独立线程，而是协议层共享状态面。

它通过：

1. `player snapshot`
2. `player event`

同步更新，然后被：

1. `SOAP` 查询
2. `GENA LastChange`

共同读取。

这避免了：

1. 协议层自己再维护另一份运行时状态
2. 查询和事件各算各的

## 6. render 并发边界

当前 render 相关边界是：

1. 主线程拥有前台页面和 render context
2. backend 不直接拥有前台 UI 生命周期
3. backend 只暴露窄 render 接口

## 7. 当前原则

当前线程设计坚持：

1. ownership 清晰优先于“多开线程”
2. 先保证状态只有一份真写入来源
3. 不为“可能未来需要”提前复杂化

## 8. 当前后续方向

当前并发层面不优先加更多线程，而是：

1. 继续稳定 mixed transport
2. 继续改善控制端同步兼容
3. 在工具链和完整后端成熟后，再评估是否拆更细的 backend/render 线程
