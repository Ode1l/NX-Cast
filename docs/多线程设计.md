# 多线程设计

本文档整理了当前对 NX-Cast（DLNA DMR + 后续多协议）线程模型的讨论结论。

## 1. 目标与约束

- 项目重点不是高并发吞吐，而是协议之间的正确性和可维护性。
- 需要保证：同一时刻只有一个协议会话控制播放器，不出现状态错乱。

## 2. MVP 线程模型（当前建议）

- `发现/控制线程`（1个）
  - 负责 SSDP、HTTP（`device.xml`/SCPD）、SOAP 控制命令接入。
  - 持续常驻监听，不因播放开始而停止，也不需要“重启发现线程”。
- `播放线程`（1个）
  - 负责拉流、解码、渲染。
  - 使用状态机驱动播放行为。

## 3. 播放状态机

建议最小状态：

- `IDLE`
- `READY`
- `PLAYING`
- `PAUSED`
- `STOPPED`
- `ERROR`

状态变更原则：

- 只允许播放线程修改播放状态。
- 协议线程（DLNA/AirPlay）通过消息队列发送 `SetURI/Play/Pause/Stop/Seek`，不直接改状态。

## 4. 多协议扩展设计

未来支持 AirPlay 等协议时，不建议“每个协议一套播放线程”，建议结构：

- 各协议各自有控制入口（可独立线程）。
- 中间增加 `会话仲裁层`（ownership arbitration）。
- 播放层仍保持一个统一播放线程（协议无关）。

## 5. 控制权与抢占规则

建议固定三条规则：

- 单一播放所有权：`owner = NONE | DLNA | AIRPLAY`
- 状态单写：播放状态只由播放线程写
- 显式抢占策略（不要隐式覆盖）：
  - `reject_if_busy`（MVP 推荐，最稳）
  - 或 `force_preempt`（先 stop 旧会话，再切新会话）

建议补充会话字段：

- `owner_protocol`
- `session_id`
- `generation`（每次切会话递增，旧命令自动失效）

## 6. 关于 `select()`

`select()` 是 I/O 多路复用：一个线程同时等待多个 socket 的可读/可写事件。

- `ret > 0`：有 fd 就绪
- `ret == 0`：超时
- `ret < 0`：错误

在本项目中可用于在单个控制线程中同时监听 SSDP/HTTP/SOAP 相关 socket，保持实现简单可控。

## 7. 当前结论（一句话）

先落地“2线程 + 统一播放状态机 + 会话所有权仲裁”，再在此基础上平滑扩展多协议。
