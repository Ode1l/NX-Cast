# 多线程设计

本文档只讨论当前 `NX-Cast` 的并发模型，不再保留早期“未来建议线程模型”的写法。

---

## 1. 目标

当前线程模型需要保证三件事：

1. 协议入口不能直接阻塞在真实播放重操作上。
2. 播放状态只能有一个真实写入来源。
3. 发现、控制、播放、事件推送彼此解耦，但边界清晰。

---

## 2. 当前线程组成

当前运行时已经稳定存在这 4 类线程：

1. `SSDP` 线程
   - 文件：[ssdp.c](/Users/ode1l/Documents/VSCode/NX-Cast/source/protocol/dlna/discovery/ssdp.c)
   - 负责：监听 `M-SEARCH`、回复 `device/service-type` 查询
2. `HTTP` 线程
   - 文件：[http_server.c](/Users/ode1l/Documents/VSCode/NX-Cast/source/protocol/http/http_server.c)
   - 负责：统一接收 `device.xml / SCPD / SOAP / SUBSCRIBE / UNSUBSCRIBE`
3. `player owner` 线程
   - 文件：[session.c](/Users/ode1l/Documents/VSCode/NX-Cast/source/player/core/session.c)
   - 负责：串行执行 `SetURI / Play / Pause / Stop / Seek / Volume / Mute`
4. `GENA event` 线程
   - 文件：[event_server.c](/Users/ode1l/Documents/VSCode/NX-Cast/source/protocol/dlna/control/event_server.c)
   - 负责：向订阅者发送 `NOTIFY` 和 `LastChange`

说明：

1. 当前没有独立的“`libmpv` 专用线程”。
2. `libmpv` 的事件循环由 `player owner` 线程定期 `pump_events()`。
3. 这意味着真实播放和 `mpv` 事件处理仍然由一个 owner 线程串行拥有。

---

## 3. 当前真实模型

当前并发模型不是“HTTP 线程直接控制 backend”，而是：

```text
HTTP/SOAP 线程
  -> player command queue
  -> player owner thread
  -> backend(libmpv/mock)
  -> player snapshot / event callback
  -> SOAP runtime state / GENA notify
```

关键点：

1. `HTTP/SOAP` 线程只负责入队和读取快照。
2. `player owner` 线程是播放命令的唯一执行者。
3. `backend` 不直接暴露给协议层。
4. `SOAP` 层和 `GENA` 层消费的是同一份真实状态变化。

---

## 4. Player 层并发边界

当前 `player` 已经具备正式的 owner-thread 模型：

1. 独立线程
2. 固定容量命令队列
3. 命令结果同步等待
4. snapshot 读取
5. 事件回调

对应文件：

1. [session.c](/Users/ode1l/Documents/VSCode/NX-Cast/source/player/core/session.c)
2. [player.h](/Users/ode1l/Documents/VSCode/NX-Cast/source/player/player.h)
3. [types.h](/Users/ode1l/Documents/VSCode/NX-Cast/source/player/types.h)

当前已经成立的结论：

1. `SOAP` 线程不会直接做 `mpv_command()`
2. `Seek`、`Pause`、`Stop` 等复杂时序已可在 owner thread 内部收敛
3. `libmpv` 事件和控制命令的顺序由 owner thread 统一拥有

---

## 5. libmpv 事件循环模型

当前 `libmpv` 不是回调式线程模型，而是 owner-thread 轮询模型：

1. `session.c` 在队列空闲时调用 `backend->pump_events()`
2. [libmpv.c](/Users/ode1l/Documents/VSCode/NX-Cast/source/player/backend/libmpv.c) 内部执行 `mpv_wait_event()`
3. 同步属性并产出 `PlayerEvent`
4. 再由 `player` 更新 snapshot 并回推给协议层

这套模型的优点：

1. ownership 清晰
2. 不需要额外 `mpv` 线程
3. 便于把命令时序和事件时序放在一个 owner 里收敛

它的边界：

1. 后续如果 `OpenGL/libmpv render API`、音频输出、硬解码都接入，owner thread 负担会继续增大
2. 那时再决定是否拆出更细的 backend/render 线程，而不是现在提前复杂化

---

## 6. 协议层并发边界

### 6.1 SSDP

`SSDP` 线程只负责发现层收发：

1. 不访问 `player`
2. 不执行 `SOAP`
3. 不参与播放状态同步

### 6.2 HTTP / SOAP

`HTTP` 线程只负责：

1. 路由 `SOAP`
2. 路由 `SCPD`
3. 路由 `SUBSCRIBE / UNSUBSCRIBE`

它不负责：

1. 长时间阻塞的播放操作
2. 主动推流
3. 自己维护真实播放状态

### 6.3 GENA event worker

事件推送采用独立 worker 的原因很简单：

1. `NOTIFY` 是网络 IO
2. 不能阻塞 `player` owner thread
3. 也不应该阻塞 `HTTP` 接入线程

---

## 7. 当前单一状态源

当前系统的状态单写原则已经成立：

1. `player snapshot` 是真实播放状态
2. `SOAP runtime state` 只是协议映射缓存
3. `GENA LastChange` 由真实状态变化驱动

因此不应再把状态理解成：

1. `SOAP` 自己维护一套
2. `player` 再维护一套

正确理解是：

1. `player` 负责真实状态
2. `SOAP` 和 `GENA` 负责协议投影

---

## 8. 当前已经解决的问题

相较于早期设计阶段，下面这些问题已经不再是现状：

1. “播放命令仍运行在 SOAP 线程里”
2. “没有命令队列”
3. “没有 `mpv` 事件循环”
4. “`SOAP` 和 `player` 完全双写状态”

这些都已经在当前代码里落地。

---

## 9. 当前仍未完成的部分

多线程设计层面仍有三件事没有完成：

1. 完整播放器后端
   - 当前路线已收敛为 `ao=hos + hwdec=nvtegra + OpenGL/libmpv render API`
2. render/backend 的最终线程模型
   - 如果未来切换到 `deko3d`，再重新收敛
3. 多协议抢占
   - 当前仍以单一 `DLNA DMR` 主链为主

---

## 10. 当前结论

一句话总结：

当前 `NX-Cast` 已经从“线程建议稿”进入“真正落地的 owner-thread + queue + event-loop 模型”，后续并发工作的重点不再是补基础线程模型，而是围绕完整音频/视频后端继续演进。
