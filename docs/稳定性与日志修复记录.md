本文档记录本轮针对 `SCPD/SSDP/SOAP` 崩溃问题的排障结论与代码修复。

## 1. 问题现象

1. `curl /device.xml` 或 `SOAP SetAVTransportURI` 时，连接被重置（`Connection reset by peer`）。
2. Switch 端应用出现“由于发生错误，软件已关闭”。
3. 手机可发现设备，但点击连接后不稳定或闪退。

## 2. 根因分析

### 2.1 SCPD 线程栈压力过大（主因）

`scpd` 线程中请求处理逻辑最初使用了较大的栈上缓冲，叠加线程栈较小，触发栈压力风险。

处理策略：

1. 大缓冲改为堆内存（`malloc/free`）。
2. `SCPD` 线程栈从原默认值提高到 `0x8000`。

### 2.2 后台线程直接打印日志导致不稳定（并发风险）

网络线程与播放 mock 线程路径中存在大量日志调用，直接输出到控制台会放大线程上下文下的不稳定风险。

处理策略：

1. 引入日志队列：后台线程只入队，不直接打印。
2. 由主线程循环中 `log_flush()` 统一输出日志。

### 2.3 SSDP 输入边界与响应范围

为避免异常 `M-SEARCH` 输入扩大风险，补充了 `ST` 值范围限制，仅响应已知目标类型。

## 3. 本轮代码改动

### 3.1 日志系统（恢复且加固）

文件：`source/log/log.h`、`source/log/log.c`、`source/main.c`

1. 新增 `log_flush()`：主线程主动刷日志。
2. `log_xxx()` 改为写入线程安全队列。
3. 新增丢包计数（队列满时统计并输出警告）。
4. 在主循环内调用 `log_flush()`，退出前再 flush 一次。
5. 日志策略调整为“包级摘要日志”：
   - 仅记录收包/发包摘要（模块、端点、字节数、状态）。
   - 不再打印完整 XML/报文正文，避免大 payload 影响实时性。

### 3.2 SCPD 稳定性修复

文件：`source/protocol/dlna/description/scpd.c`

1. 大型请求/响应缓冲改为堆分配（避免压线程栈）。
2. 线程栈常量提升为 `SCPD_THREAD_STACK_SIZE = 0x8000`。
3. 补充注释说明本次修复背景。
4. 恢复关键请求日志（`SOAP handled`、`Served`、未知路径）。

### 3.3 SSDP 稳定性修复

文件：`source/protocol/dlna/discovery/ssdp.c`

1. 线程栈常量提升为 `SSDP_THREAD_STACK_SIZE = 0x8000`。
2. 仅对已知 `ST` 值响应（`ssdp:all`、`upnp:rootdevice`、`device_type`、`uuid`）。
3. `sendto` 失败时记录告警日志。
4. 补充注释说明边界控制目的。

### 3.4 Player Mock 日志恢复

文件：`source/player/mock.c`

恢复以下动作日志（通过新日志队列输出）：

1. `init/deinit`
2. `set_uri`
3. `play/pause/stop`
4. `seek`
5. `set_volume/set_mute`

## 4. 验证结果

1. `GET /device.xml` 返回 `HTTP 200` 且 XML 正常。
2. `SetAVTransportURI` 单独请求返回 `HTTP 200`。
3. `scripts/dlna_soap_smoke.sh` 全流程通过（`PASS`）：

## 5. 后续建议

1. 若后续日志量继续变大，可把 `LOG_QUEUE_CAPACITY` 从当前 `512` 继续调高。
2. 长期建议将网络线程日志分级（默认 `INFO` 降噪，异常走 `WARN/ERROR`）。
3. 后续接入真实播放器后，继续复用当前日志队列机制，避免回到“后台线程直接打印”模式。
