# 源兼容性验证

## 1. 目标

当前 `NX-Cast` 的控制层已经能把 `SetAVTransportURI / Play / Pause / Stop / Seek / GetPositionInfo / GetTransportInfo` 传到 `player-libmpv`。

现阶段首先要验证的，不是 SOAP 是否还能继续扩功能，而是下面这件事：

1. `libmpv` 是否能成功识别并打开控制端传来的媒体 URL

只要媒体真实打开成功，下面这些控制一般都会随之恢复：

1. 手机端进度条更新
2. 快进快退
3. 进度条拖拽
4. 暂停 / 恢复播放
5. `GetPositionInfo` 返回真实时间
6. `GetTransportInfo` 返回真实播放状态

当前已经确认的一类失败日志是：

1. `[player-libmpv] event END_FILE reason=error error=unrecognized file format`
2. `[player-libmpv] mpv[cplayer/error] Failed to recognize file format.`

这说明问题不在 SOAP 参数解析，而在媒体源兼容性。

---

## 2. 验证思路

源兼容性验证的目标，是把问题拆成三层：

1. `DLNA/SOAP` 是否把 URL 正确交给 player
2. `libmpv` 是否能真正打开这个 URL
3. 控制端给出的 URL 属于哪一类源

因此验证顺序应固定为：

1. 先用“已知可直链播放”的媒体 URL 建立基线
2. 再用手机真实投屏给出的 URL 做对比
3. 最后判断失败是不是 URL 类型本身导致的

不要一开始就拿手机随机投屏 URL 做唯一测试源，否则无法区分：

1. 是 `player-libmpv` 实现有问题
2. 还是这个 URL 本身需要特殊 Header / Cookie / Referer
3. 还是它根本不是一个标准直链媒体地址

## 4. 最小验证流程

### Step 1. 启动最新程序并接日志

```sh
./scripts/run_nxlink.sh
```

通过标准：

1. 日志中出现 `player-libmpv` 初始化
2. 日志中出现 `remote TCP log enabled`
3. 运行日志写入 `logs/run_nxlink-*.log`

关键日志示例：

```text
[INFO] [player] auto resolve libmpv_available=1 mock_available=1
[INFO] [player-libmpv] init mode=control-only ao=null vo=null
[INFO] [player] init backend=libmpv
```

如果这里仍然不是 `backend=libmpv`，后续验证全部无效。

### Step 2. 用已知可控 URL 走一遍固定 SOAP 流程

使用 `dlna_soap_smoke.sh` 时，明确指定 `MEDIA_URI`：

```sh
HOST=192.168.1.7 PORT=49152 \
MEDIA_URI='http://60.29.124.66:6080/hls/26/index.m3u8' \
SEEK_TARGET='00:00:00' \
./scripts/dlna_soap_smoke.sh
```

说明：

1. `HOST` 和 `PORT` 是 Switch 的 HTTP 服务地址
2. `MEDIA_URI` 是你要验证的具体媒体 URL
3. `SEEK_TARGET` 是脚本里用来测试 `Seek` 的目标时间

通过标准：

1. `SetAVTransportURI` 返回 `200`
2. `Play` 返回 `200`
3. `GetTransportInfo` 返回 `PLAYING`
4. `GetPositionInfo` 中 `RelTime` 不再一直是 `00:00:00`
5. `Seek` 返回 `200`
6. `Pause` 返回 `200`
7. `Stop` 返回 `200`

如果 `Play` 表面成功，但日志里很快出现：

```text
[player-libmpv] event END_FILE reason=error error=unrecognized file format
[player-libmpv] mpv[cplayer/error] Failed to recognize file format.
```

那么这不是控制流成功，而是“上层收到了命令，下层没打开媒体”。

## 5. 如何选测试源

源兼容性验证必须分组做，不要只测一种 URL。

建议按下面四类准备测试源：

### A. 纯直链静态媒体

特征：

1. URL 直接指向 `.mp4` / `.mp3`
2. 不依赖 Cookie
3. 不依赖 Referer
4. 浏览器打开就是媒体本身，不是 HTML 页面

作用：

1. 验证 `player-libmpv` 基本拉流能力
2. 验证 SOAP 的 `Play/Pause/Seek/GetPositionInfo` 基础闭环

这类源是基线。

### B. 带签名参数的媒体 URL

特征：

1. URL 很长
2. 带 `e=...`、`token=...`、`sign=...` 一类参数
3. 可能有时效性

作用：

1. 验证 `libmpv` 是否支持这类临时直链
2. 排除“签名过期”导致的假兼容问题

### C. 需要 Header 的媒体 URL

特征：

1. 没有 `Referer` / `User-Agent` 就返回 HTML 或错误页
2. 手机 App 内可播，但裸 URL 不一定能播

这类源是当前 Bilibili 风格 URL 最值得怀疑的类型。

### D. 非文件型流媒体

特征：

1. `m3u8`
2. DASH / 分片流
3. 重定向多层跳转

作用：

1. 验证 `libmpv` 的协议支持边界
2. 区分“普通 HTTP 文件流”和“流媒体播放”的差异

## 7. 如何看日志

### 成功加载的典型信号

如果一个源是兼容的，日志中应尽量出现：

```text
[player-libmpv] event START_FILE
[player-libmpv] event FILE_LOADED
```

然后后续应看到：

1. `GetTransportInfo` 返回 `PLAYING`
2. `GetPositionInfo` 的 `RelTime` 逐渐增加
3. `Pause` 不再返回 `701`
4. `Seek` 调用开始稳定出现且返回成功

### 失败加载的典型信号

如果源不兼容，通常会出现：

```text
[player-libmpv] event START_FILE
[player-libmpv] event END_FILE reason=error error=...
[player-libmpv] mpv[.../error] ...
```

典型后果：

1. `Pause` 返回 `701 Transition not available`
2. `GetPositionInfo` 一直停在 `00:00:00`
3. 手机端进度条始终不动
4. 控制端不稳定发 `Seek`

## 8. 如何搭建测试脚本

现阶段建议不要立刻写一个很大的“万能脚本”，而是搭两层脚本。

### 8.1 第一层：复用现有 `dlna_soap_smoke.sh`

这一层已经够用来验证：

1. 固定 URL 能否跑完整控制流程
2. `Play/Pause/Seek` 是否成功
3. `GetPositionInfo/GetTransportInfo` 是否能正常返回

使用方式：

```sh
HOST=192.168.1.7 PORT=49152 \
MEDIA_URI='http://your-test-url/test.mp4' \
SEEK_TARGET='00:00:15' \
./scripts/dlna_soap_smoke.sh
```

补充：

1. 当 `MEDIA_URI` 是 `m3u8` 时，脚本会自动使用更长的等待窗口
2. 等待 `PLAYING` 期间会额外打印 `CurrentTransportState / RelTime / TrackDuration`
3. 可以通过 `HLS_STATE_WAIT_SECONDS` 单独调大 HLS 等待时间
4. `player-libmpv` 会额外输出三类 HLS 诊断：
- `diag`
- `diag detail`
- `cache_state`
5. HLS 等待 `PLAYING` 时，脚本会先查 `GetPositionInfo` 再查 `GetTransportInfo`，并在出现非零 `TrackDuration` 后做一小段 `grace retry`

这三类日志的判定作用是：

1. `diag`
- 看状态机是否从 `LOADING` 前进到 `BUFFERING / PLAYING`

2. `diag detail`
- 看 `mpv` 当前打开的是 playlist 还是实际 segment
- 重点字段是 `demuxer / stream_open / stream_path / path`

3. `cache_state`
- 看缓存是否真的开始增长
- 如果长期 `cache_ms=0 / cache_speed_bps=0`，通常说明还没真正进入分片读取阶段

### 8.2 第二层：做一个“多 URL 矩阵测试脚本”

推荐未来新增：

```text
scripts/source_compat_matrix.sh
```

职责：

1. 读取一组测试 URL
2. 逐个调用 `dlna_soap_smoke.sh`
3. 为每个 URL 生成一份结果摘要
4. 最后输出矩阵结论

推荐输入格式：

```text
# label|url|seek_target
A1|http://example.com/test.mp4|00:00:10
A2|http://example.com/test.mp3|00:00:05
B1|http://example.com/signed.mp4?...|00:00:15
C1|http://example.com/header-required.mp4|00:00:10
```

推荐脚本结构：

```sh
#!/usr/bin/env bash
set -euo pipefail

HOST="${HOST:-192.168.1.7}"
PORT="${PORT:-49152}"
CASE_FILE="${1:-scripts/source_cases.txt}"

while IFS='|' read -r label media_uri seek_target; do
  [ -z "$label" ] && continue
  [ "${label#\#}" != "$label" ] && continue

  echo "[matrix] case=${label} uri=${media_uri}"
  HOST="$HOST" PORT="$PORT" MEDIA_URI="$media_uri" SEEK_TARGET="$seek_target" \
    ./scripts/dlna_soap_smoke.sh || echo "[matrix] case=${label} failed"
done < "$CASE_FILE"
```

这类脚本的价值不是替代日志，而是把“不同 URL 是否兼容”做成可重复实验。

## 10. 如何判定下一步改什么

### 情况 A：基线直链 MP4 可以正常播放

说明：

1. `player-libmpv` 基本链路没问题
2. SOAP 进度、暂停、拖拽逻辑可继续沿用
3. 当前应重点做“特定来源兼容”

下一步应改：

1. `mpv` HTTP 参数
2. Header / Referer / User-Agent 支持
3. 针对特定源做兼容实验

### 情况 B：基线直链 MP4 也失败

说明：

1. 不是来源兼容问题
2. 是 `player-libmpv` Step 1 还没稳定

下一步应改：

1. `mpv` 初始化参数
2. backend 状态机
3. `FILE_LOADED / END_FILE` 同步逻辑

### 情况 C：只有手机投屏 URL 失败

说明：

1. DMR 控制链路已基本可用
2. 失败集中在“控制端提供的这类 URL”
3. `Seek / Pause / 进度条` 的异常是次生问题

下一步应改：

1. 源兼容策略
2. Header 注入
3. URL 预处理

## 12. 2026-03-31 追加结论

这一轮 `https mp4 / https m3u8` 验证后，已经可以把问题再拆细一层。

### 12.1 `SEEK_TARGET=00:00:00` 不再作为真实 Seek

从这一天开始，`dlna_soap_smoke.sh` 的规则应视为：

1. `SEEK_TARGET=00:00:00`
2. `SEEK_TARGET=""`

都表示：

1. 跳过 `Seek`
2. 只验证 `SetURI / Play / Pause / Stop / GetTransportInfo / GetPositionInfo`

这样做的原因很直接：

1. “跳到 0 秒”对能力验证没有价值
2. 它会把“媒体尚未 ready”误判成“Seek 失败”

因此：

1. `seek=0` 的 `PASS` 只代表“非 Seek 控制链路通过”
2. 不代表真正的 `Seek` 已经通过

### 12.2 `seek=2s` 失败的真实含义

当前 `seek=2s` 失败时，最常见的是：

1. SOAP `701 Transition not available`

这不再应该简单理解成“Seek 功能坏了”，而应该理解成：

1. 当前媒体还没有进入真实可 seek 状态
2. 或当前媒体源本身就不可 seek

因此当前实现已经补了两个判断：

1. `player-libmpv` 只在真实 `FILE_LOADED` 之后才认为媒体 ready
2. SOAP `Seek` 只在 `player_is_seekable()` 为真时才放行

也就是说：

1. 以后 `Seek 701`
2. 更接近真实“媒体未 ready / 不可 seek”

而不是之前那种“只要有 URI 就尝试 seek”的假动作。

### 12.3 `https mp4` 目前的判断边界

当前 `https mp4` 不能再被宽松地判定为“已经好了”。

日志里真正应该关注的是：

1. 是否出现 `FILE_LOADED`
2. `GetTransportInfo` 是否在此之后才进入 `PLAYING`
3. `GetPositionInfo` 是否开始递增
4. `Seek` 是否在 loaded 之后成功

如果日志只有：

1. `set_uri`
2. `play reloaded=0`

但没有：

1. `FILE_LOADED`

那么只能说明：

1. “播放请求没有再二次 reload”

不能说明：

1. 文件已经真正 ready
2. 更不能说明它已经可 seek

### 12.4 `https m3u8` 当前是更深一层的问题

这轮日志已经证明：

1. `playlist` 本身可以打开
2. 真正失败点在首个 HLS 分片加载

典型日志形态是：

1. `tls: mbedtls_ssl_handshake returned -0x7280`
2. `hls: Error when loading first segment`
3. `avformat_open_input() failed`

其中 `-0x7280` 对应：

1. `MBEDTLS_ERR_SSL_CONN_EOF`

这说明当前问题不是：

1. SOAP 路由
2. 普通 URL 解析
3. HLS playlist 解析

而更像是：

1. HLS 分片 HTTPS/TLS 握手被对端关闭

因此这类问题应归入：

1. `FFmpeg/mpv + TLS/HLS` 兼容问题

而不是继续归咎于 DMR 控制层。

### 12.5 当前必须坚持的判定原则

后续分析日志时，必须坚持下面这组规则：

1. `FILE_LOADED` 之前，不把媒体视为真正 ready
2. `seekable=false` 时，不把 `Seek 701` 视为异常误判
3. `seek=0` 的成功，不纳入 Seek 兼容性结论
4. `https mp4` 和 `https m3u8` 的结论必须分开写

当前必须遵守的现阶段结论是：

1. 本地 `http faststart mp4`：可作为稳定基线
2. `https mp4`：曾通过，但仍应以当前分支、当前日志为准，不把旧结果当永久结论
3. `https m3u8`：当前不能再写成“已经稳定通过”，最新日志已出现真实 `TLS/HLS` 失败
4. `bilibili`：当前不能再写成“只剩 header 微调”，最新日志已经说明它属于 `vendor-sensitive source`

### 12.6 2026-03-31 最新结论

截至 `2026-03-31` 最新一轮日志，当前必须接受下面三个事实：

1. `https m3u8` 仍可能在真实 `TLS/HLS` 路径失败
2. `bilibili` 仍可能在真实来源访问阶段返回 `HTTP 403`
3. 继续按单个 bug 修补，已经不足以解释当前问题
