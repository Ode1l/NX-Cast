# 源兼容性

## 1. 文档目的

这份文档不再只做“验证记录”，而是作为当前 `NX-Cast` 的源兼容性说明。

它回答四个问题：

1. 当前项目到底在哪一层判断“能不能播”
2. 什么问题属于 `SOAP`，什么问题属于 `player/ingress/policy`
3. 为什么不同站点差异很大
4. 当前已经验证通过和仍未解决的源有哪些

---

## 2. 当前系统边界

当前媒体链路是：

1. `DLNA/SOAP` 接收 `SetAVTransportURI / Play / Pause / Stop / Seek`
2. `player core` 维护统一状态机与 snapshot
3. `ingress` 负责 URI / metadata 解析、站点识别、格式识别、资源选择、策略注入
4. `backend/libmpv` 负责真实拉流、状态同步、render 输出

因此，源兼容性问题一般不再优先怀疑 `SOAP` 是否“能调用到 player”，而是重点看：

1. `ingress` 是否识别对了站点与格式
2. `policy` 是否给了足够的请求上下文
3. `libmpv` 请求的是否是真正可播的 URL

---

## 3. 第一性原则

### 3.1 控制协议不等于源兼容

`DLNA` 统一的是控制动作，不统一内容站的授权方式。

即使下面这些都成功：

1. `SetAVTransportURI`
2. `Play`
3. `GetTransportInfo`
4. `GetPositionInfo`

也不代表站点一定会让接收端成功拉到真实媒体分片。

### 3.2 能解码不等于能取流

`mp4 / hls / ts / flv` 的格式支持只是基础能力。

真正决定成败的常常是：

1. `User-Agent`
2. `Referer`
3. `Origin`
4. `Cookie`
5. 额外鉴权 header
6. URL 时效和签名
7. sender 给的是最终可播 URL，还是仅对它自身有效的中间 URL

### 3.3 站点源没有“一套完美通用方案”

市售盒子通常也不是只靠一套纯通用策略通吃所有站点。

更现实的工程形态是：

1. 通用播放器底座
2. 站点分类识别
3. 重点站点的 `UA / Referer / Cookie / resource select` 定向适配
4. 必要时由本地 App / 系统能力接管，而不是继续走 generic DMR

`NX-Cast` 当前已经具备通用底座和第一版站点适配框架，但大站兼容仍然是“站点策略工程”，不是纯解码工程。

---

## 4. sender UA 与播放 UA

这是当前兼容性里最容易混淆的概念。

### 4.1 sender UA

`sender UA` 是控制点发给 DMR 的 HTTP 请求头，例如：

1. `UPnP/1.0 IQIYIDLNA/iqiyidlna/NewDLNA/1.0`

它表示：

1. 谁在控制接收端
2. 哪个手机投屏组件在发 SOAP

它属于：

1. `手机 App -> Switch DMR`

### 4.2 播放 UA

`播放 UA` 是 `Switch/mpv` 去请求：

1. `mp4`
2. `m3u8`
3. `ts`
4. 其他媒体资源

时发送给站点/CDN 的 `User-Agent`。

它属于：

1. `Switch/mpv -> 媒体站点/CDN`

### 4.3 设计原则

两者语义不同，不应默认等同。

当前项目采用的原则应是：

1. 默认播放 UA 当前使用稳定的浏览器兼容 UA
2. `sender UA` 只采集、记录、必要时作为调试依据
3. 不应直接用 `sender UA` 覆盖播放 UA
4. 如果某个站点后续验证明确依赖特定 UA，再做站点级 override

原因：

1. `sender UA` 说明控制端身份
2. `播放 UA` 才是资源站点实际拿来做授权/风控/兼容判断的上下文

补充原则：

1. 当前默认浏览器 UA 只是工程上的保守基线，不应被视为最终形态
2. 通用策略的目标应是“中性客厅设备 / 市售电视盒子风格”的播放策略
3. 这个目标不等于全局伪装成 `Apple TV`
4. `Apple TV` 更接近 `AirPlay + App/System 媒体栈`，不是 generic `DLNA DMR` 的标准样板
5. 真正接近市售盒子的方式，应是“中性默认策略 + 重点站点专项 override”，而不是“一个 UA 通吃所有站点”

---

## 5. 当前兼容性架构

### 5.1 ingress

`ingress` 当前负责：

1. 站点识别：`bilibili / iqiyi / mgtv / youku / qq-video / cctv`
2. 格式识别：`mp4 / hls / dash / flv / ts`
3. metadata 中 `res/protocolInfo` 候选解析与打分选择
4. 构造 `PlayerMedia`

### 5.2 policy

`policy` 当前负责：

1. 默认播放 UA、超时、probe 参数
2. HLS startup profile
3. vendor-sensitive URL 的 `Referer / Origin / header_fields`
4. sender 请求上下文的合并通道

当前 sender 请求上下文已具备通道，但实际能否生效，取决于控制点是否真的在请求里带了这些信息。

当前默认策略说明：

1. 默认策略仍然是浏览器兼容播放 UA
2. 这只是当前阶段的通用基线，不代表最终目标
3. 后续更合理的方向，是把默认策略收敛成“中性电视盒子/客厅设备”风格
4. 站点专项策略仍然保留，例如 `mgtv` 当前已开始尝试单独使用 `ImgoTV` 风格播放 UA

### 5.3 backend/libmpv

`libmpv` 当前负责：

1. 真正拉取 URL
2. 状态机同步
3. startup / cache / hls 诊断
4. 视频 render

因此日志中一旦出现：

1. `HTTP error 400`
2. `Failed to open`
3. `Failed to open segment`
4. `avformat_open_input() failed`

优先判断为“源请求上下文问题”而不是“SOAP 没通”。

## 6. 当前已验证结果

以下结论基于近期真实日志与实机验证。

### 6.1 已通过

1. 直链 `mp4`
   结论：主链已通，`SOAP -> player -> libmpv -> render` 成立

2. 中国 `live HLS`
   结论：可播，`Pause/Stop` 正常，`Seek` 依赖源能力  
   限制：启动慢，且 live 源经常不可 seek

3. `bilibili live`
   结论：可播

4. `bilibili` 电影 / 普通视频
   结论：近期已通过  
   经验：完整 `SOAP` 输出、metadata 返回和 vendor policy 对控制端兼容性影响很大

### 6.2 未通过或不稳定

1. `iqiyi mp4`
   最新现象：站点直接返回 `HTTP 400`

2. `iqiyi hls`
   最新现象：playlist 路径能进入，但首个 `ts` 分片打开失败，最终 `avformat_open_input() failed`

当前对 `iqiyi` 的判断是：

1. 不是 `SOAP` 失败
2. 不是 generic `hls/mp4` 能力缺失
3. 更像站点请求上下文不足

### 6.3 本地加速与混合源

这里的“混合源”是指：

1. 同一控制端有时给公网直链
2. 有时给手机本地代理 URL
3. 有时 metadata 声称是 `mp4`，实际拉流却进入 `hls`

当前项目对这类源的能力边界是：

1. 已经可以识别手机本地代理 HLS，并与公网 HLS 区分
2. 已经可以按每次 `SetAVTransportURI` 重新分类，不会把“同一家 App 的不同链路”硬套成同一种源
3. 已经可以通过 `format_hint/local_proxy` 把这类流打进独立日志与策略入口
4. 但还没有做“本地代理 transport”的专用重试、保活、segment 恢复和 fallback

因此，当前对混合源的结论应是：

1. 能接住
2. 能分类
3. 部分可播
4. 但还不能宣称已经稳定支持“带本地加速功能的投屏”

近期样例可归纳为：

1. 腾讯视频
   现象：不开启本地加速时，可进入 `qq-video` 路径并部分可播，但首次启动、位置同步和拖动时序仍不够稳定

2. 央视网
   现象：经常给出手机本地代理 `m3u8`
   判断：问题更像手机代理 HLS 的后续 segment 失败，而不是 render 或 `SOAP` 故障

3. 优酷
   现象：metadata 看起来像 `mp4`，实际 `demuxer/file_format` 进入 `hls`
   判断：更像站点网关/代理流，不应继续按纯 `mp4` 心智处理

工程上的原则是：

1. “本地加速源”不等于普通公网 HLS
2. “网关伪装成 mp4 的 HLS”也不等于普通 `mp4`
3. 这两类都应被视为独立 transport 形态，而不是继续只在 vendor policy 里打补丁

后续如果继续做这一方向，合理的演进路径应是：

1. 为 `local_proxy` 单独定义 transport policy
2. 为手机代理 HLS 增加超时、重试、segment 失败诊断
3. 必要时将“手机代理源”和“公网真实源”分成两条运行时策略
4. 将“广告 / 主片 / 本地代理 / 公网直链”视为可切换会话，而不是单一源

## 7. 如何判定问题在哪一层

### 7.1 SOAP 层问题

通常会表现为：

1. `SetAVTransportURI / Play / Pause / Stop` 直接 fault
2. `GetTransportInfo / GetPositionInfo` 返回异常
3. 控制端 UI 进度条、拖动、播放态异常，但底层 URL 其实能播

### 7.2 ingress / policy 问题

通常会表现为：

1. vendor 识别错误
2. format 识别错误
3. metadata 候选资源选择错误
4. `Referer / Origin / UA / Cookie` 不足

日志关注：

1. `resolve_media`
2. `selected_resource`
3. `source_detail`
4. `metadata_res`
5. `runtime_overrides_detail`

### 7.3 backend / 取流问题

通常会表现为：

1. `START_FILE` 后没有 `FILE_LOADED`
2. `HTTP error 400/403`
3. `Failed to open`
4. `Failed to open segment`
5. `END_FILE reason=error`

这类现象往往不是控制协议问题，而是 URL / header / 站点策略问题。

---

## 8. 当前日志判读规则

### 8.1 一条源“真的成功”

至少要看到：

1. `SetAVTransportURI -> 200`
2. `Play -> 200`
3. `FILE_LOADED` 或真实进入 `PLAYING`
4. `GetPositionInfo` 的时间递增
5. `Pause/Stop` 与状态切换一致

### 8.2 一条源“只是控制成功，播放失败”

常见表现：

1. `SetAVTransportURI -> 200`
2. `Play -> 200`
3. `GetTransportInfo` 仍在 `TRANSITIONING/STOPPED`
4. `mpv` 报 `HTTP error 400/403`
5. `Failed to open` / `Failed to open segment`
6. `END_FILE reason=error`

这说明：

1. 控制链路没问题
2. 媒体源没真正打开

---

## 9. iqiyi 当前结论

截至当前项目状态，`iqiyi` 的关键事实是：

1. sender 给到的 metadata 很弱，通常只有 bare DIDL-Lite，没有 `res/protocolInfo`
2. 当前日志里常常没有 `Cookie`
3. `m3u8` 本身进入了 HLS 路径，不代表 segment 会成功
4. 失败经常发生在首个 `ts` 分片请求，而不是 manifest 解析

因此，后续 `iqiyi` 兼容的重点应是：

1. 继续保留并完善请求上下文通道
2. 保持播放 UA 为浏览器兼容 UA，而不是直接使用 sender UA
3. 按真实日志验证是否还缺 `Cookie` 或额外 header

---

## 10. 后续工作建议

按优先级：

1. 继续按站点扩展 `vendor-sensitive policy`
2. 保持“中性默认策略 + 站点专项 override”的方向，不走“全局 Apple TV 化”
3. 继续观察 `iqiyi` 是否能从 sender 侧拿到更多上下文
4. 在默认策略稳定后，再考虑把浏览器兼容默认 UA 收敛成更像市售盒子的中性播放策略
5. 只有在再次遇到控制端异常时，才回头怀疑 `SOAP`

这份文档的作用，是把“源不兼容”和“控制不通”分开，避免后续排查反复回到已经成立的链路上。
