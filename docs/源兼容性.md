# 源兼容性

本文档记录 `NX-Cast` 当前如何理解“源兼容性”。

## 1. 当前问题边界

源兼容性不再被简单理解为：

1. `SOAP` 能不能发到 player
2. `mpv` 能不能解这个格式

当前真正影响结果的是三层：

1. 标准输入是否被正确建模
2. transport 是否被正确识别
3. 打开策略是否足够稳

## 2. 第一性原则

### 2.1 控制协议不等于源兼容

`DLNA` 统一的是控制动作，不统一站点授权方式。

所以：

1. `SetAVTransportURI` 成功
2. `Play` 成功

不等于站点一定允许后续真实拉流。

### 2.2 解析与策略必须分开

先回答：

1. 这是什么源

再回答：

1. 怎么更稳地打开它

这也是当前 `ingress` 结构收口的原因。

### 2.3 vendor 只做加法

当前原则是：

1. 先按标准证据解析
2. `vendor` 只做 additive hint
3. 不允许 vendor 反过来推翻已经明确的标准解析结果

## 3. 当前标准输入模型

当前标准输入包括：

1. `CurrentURI`
2. `CurrentURIMetaData`
3. sender 请求头
4. metadata `res/protocolInfo`
5. URL preflight 结果

它们先统一进入：

- `IngressModel`

然后再 materialize 成最终 `PlayerMedia`。

## 4. 当前 transport 分类

当前显式 transport：

1. `http-file`
2. `hls-direct`
3. `hls-local-proxy`
4. `hls-gateway`

这比“只按 vendor 或后缀判断”更重要。

## 5. sender UA 与播放 UA

当前明确区分：

1. `sender UA`
   - 控制端身份
2. `播放 UA`
   - 接收端向媒体站点发请求时的 UA

这两者不应默认等同。

当前原则：

1. `sender UA` 主要用于观察和少量 hint
2. 播放 UA 应保持独立
3. 站点 override 只在明确必要时做

## 6. 当前系统性结论

经过多轮验证后，当前可以确定：

1. 许多兼容性收益来自通用 DMR 完整度，而不只是站点 hack
2. `Macast` 这类项目的价值主要在完整协议状态和标准行为
3. 混合源问题通常首先是 transport 问题
4. `iqiyi` 这类仍可能超出纯 generic DMR 的能力边界

## 7. 当前各类源的工程判断

### 7.1 bilibili

通常更接近“generic DMR 可兼容”的源，但仍可能受到：

1. URL 时效
2. signed URL
3. 请求上下文

影响。

### 7.2 mgtv-family

兼容收益更多来自：

1. 正确 family 识别
2. 更合理的播放策略
3. transport 稳定性

### 7.3 qq-video

当前更像：

1. 播放链基本可通
2. 进度同步强依赖控制端行为
3. 混合源切换和本地加速模式更值得关注

### 7.4 youku / cctv

当前更偏 transport 问题：

1. `HLS gateway`
2. `local_proxy`
3. 慢启动
4. 分片失败

### 7.5 iqiyi

当前仍然最像：

1. 请求上下文不足
2. 站点授权链更强

它不应被当成通用 DMR 架构是否正确的唯一判据。

## 8. 当前重点

当前源兼容性的重点不是继续堆站点特例，而是：

1. 完成标准输入建模
2. 完成 transport 级行为
3. 继续提高控制端互操作
4. 最后再针对少数站点做最小必要 override
