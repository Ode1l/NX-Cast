# 源兼容性

本文档记录 `NX-Cast` 当前如何理解“源兼容性”。

## 1. 当前问题边界

源兼容性不再被简单理解为：

1. `SOAP` 能不能发到播放器
2. `mpv` 能不能解这个格式

当前真正影响结果的是：

1. 控制端给出的 URL 是否真实可用
2. 请求上下文是否足够让接收端继续拉流
3. `libmpv/ffmpeg` 是否能自动识别并打开该媒体
4. 协议状态能否正确反映真实播放进度和失败原因

## 2. 第一性原则

### 2.1 控制协议不等于源兼容

`DLNA` 统一的是控制动作，不统一站点授权方式。

所以：

1. `SetAVTransportURI` 成功
2. `Play` 成功

不等于站点一定允许后续真实拉流。

### 2.2 播放链保持薄

当前播放器路径的原则是：

1. 协议层拿到 URL
2. 直接交给 renderer
3. renderer 交给 `libmpv`
4. 由 `libmpv/ffmpeg` 自动探测和打开

兼容性问题首先应该在真实请求链和真实运行时状态上定位，而不是先发明额外的本地推断层。

### 2.3 运行时状态必须回流

兼容性判断不能只看“命令发出去了”，还要看：

1. `time-pos`
2. `duration`
3. `pause`
4. `seekable`
5. EOF / error

这些真实运行时状态有没有从 `libmpv` 正确回流到协议状态。

## 3. 当前系统性结论

经过多轮验证后，当前可以确定：

1. 许多兼容性收益来自通用 `DMR` 完整度，而不只是站点 hack
2. `Macast` 这类项目的价值主要在完整协议状态和播放器事件同步
3. 很多失败其实是 URL 时效、header、cookie、referer 一类请求上下文问题
4. 少数站点可能仍超出纯 generic `DMR` 的能力边界

## 4. 当前工程判断

### 4.1 对常见直链媒体

如果控制端给的是标准 `http(s)` 音视频 URL，当前模型通常就是：

1. 直接交给 `libmpv`
2. 让其自动识别容器和流
3. 再根据运行时状态决定是否兼容

### 4.2 对 `HLS` 和站点链路

如果控制端给的是 `m3u8`、签名 URL 或强依赖上下文的媒体链路，当前更该关注：

1. URL 是否还在时效内
2. 接收端请求头是否足够
3. 控制端是否在正确时机发起 `Play / Seek`
4. 协议状态是否把失败原因正确暴露出来

### 4.3 对难兼容站点

像 `iqiyi` 一类站点，更可能是：

1. 请求上下文不足
2. 授权链更强
3. 单纯 generic `DMR` 不足以完全覆盖

它不应被当成通用架构是否正确的唯一判据。

## 5. 当前重点

当前源兼容性的重点不是继续堆站点特例，而是：

1. 保持描述层与实际能力一致
2. 保持 renderer 与协议状态同步准确
3. 在真实控制端上改善互操作
4. 只在明确必要时补最小 request-context 处理
